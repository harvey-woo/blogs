一行命令解决 AI 编程工具的 API 限流焦虑
========================================

最近不少同学找我抱怨：用 Cursor、CodePlan 之类的 AI 编程工具时，频繁碰到 **"请求过于频繁"** 的错误。

尤其是公司团队一起用的时候，百炼、通义等平台的 API 限流策略会被瞬间触发。你正写着代码，补全突然断了；或者让 AI 重构一个模块，跑到一半报错 429。体验极差。

这个问题的本质是：**AI 编程工具会在短时间内发起大量并发请求**，而很多 AI 平台的限流策略是按并发数或 QPS 来控制的。当团队共用一个 API Key 时，这个问题会被放大数倍。

## 为什么会出现限流

以百炼为例，它的 API 对并发请求有明确的限制。但 AI 编程工具（比如 Cursor）在以下场景会瞬间发起大量请求：

- 打开文件时，后台同时请求多个文件的代码补全
- 多人协作时，每个人的补全请求叠加
- AI Agent 批量读取文件、执行任务时的高频调用

这些请求如果直接打到上游 API，很容易触发限流。而限流的后果就是：**你的编程工具突然变砖**。

## 常见的应对方式

**方式一：换 Key / 换平台。** 这治标不治本，只是把问题推迟了。

**方式二：让团队成员错峰使用。** 不现实，没人愿意排队写代码。

**方式三：在客户端做限流。** 但像 Cursor 这种闭源工具，你改不了它的请求行为。

**方式四：中间加一层代理。** 这才是正解。

## rate-proxy：一行命令解决限流

我做了一个小工具 **rate-proxy**，它是一个带并发限制的 HTTP 反向代理。原理很简单：在你的 AI 编程工具和上游 API 之间加一层，把所有请求统一经过代理，代理控制并发数。

```bash
# 安装
npm install -g @cat5th/rate-proxy

# 启动代理，限制并发数为 5
TARGET=https://dashscope.aliyuncs.com CONCURRENCY=5 rate-proxy
```

然后在你的 AI 编程工具里，把 API 地址改成本地代理 `http://localhost:8080` 就行了。

无论你的工具同时发起多少个请求，代理会保证最多只有 `CONCURRENCY` 个请求到达上游，其余自动排队等待。

## 原理：一个并发闸门

rate-proxy 的核心实现只有几行代码。它基于我写的另一个库 [@cat5th/pool.js](https://github.com/harvey-woo/pool.js)，这是一个基于 ES2024 显式资源管理的轻量级资源池调度器。

```typescript
import { Pool } from '@cat5th/pool.js';
import { createServer } from 'node:http';

const pool = new Pool(CONCURRENCY);
const scheduler = pool.schedule();

const server = createServer(async (req, res) => {
    await scheduler.enqueue(async function () {
        const response = await fetch(TARGET + req.url, {
            method: req.method,
            headers: req.headers,
            body: req,
        });
        res.writeHead(response.status);
        response.body.pipeTo(res);
    });
});

server.listen(8080);
```

关键就是 `scheduler.enqueue`。它相当于一个**并发闸门**：

- 当并发数未达上限时，请求直接透传到上游
- 当并发数达到上限时，新请求进入队列等待
- 一旦有请求完成，队列中的下一个自动发出

整个过程对客户端完全透明，你的 AI 工具根本不知道中间有一层代理在控制节奏。

## 扩展场景

rate-proxy 不限于解决 AI 编程工具的限流问题。任何需要控制并发请求的场景都可以用它：

**爬虫速率控制。** 你写了一个爬虫，但目标网站对请求频率有限制。通过 rate-proxy 代理所有请求，避免被封 IP。

```bash
TARGET=https://example.com CONCURRENCY=2 rate-proxy
```

**第三方 API 代理。** 很多 SaaS 平台（如 OpenAI、飞书、钉钉）都有并发限制。用 rate-proxy 做一个本地代理，统一控制请求节奏。

**数据库连接代理。** 某些云数据库对并发连接有严格限制（比如免费 tier），通过代理避免连接数超限。

## 进阶：自定义冷却间隔

除了并发数控制，rate-proxy 底层使用的 pool.js 还支持 **CoolDown** 机制。如果你不仅需要控制并发数，还需要控制请求频率（比如每秒最多 N 次），可以通过冷却回调实现：

```typescript
const pool = new Pool({
    concurrency: 5,
    coolDown({ deliverAt, releaseAt }) {
        // 确保两次请求之间至少间隔 200ms（即每秒最多 5 次）
        const elapsed = releaseAt - deliverAt;
        const wait = Math.max(0, 200 - elapsed);
        return new Promise(resolve => setTimeout(resolve, wait));
    }
});
```

这个机制特别适合那些同时限制并发数和 QPS 的平台。

## 关于 pool.js

rate-proxy 的核心依赖是 pool.js，一个基于 ES2024 显式资源管理（`using` / `await using`）的资源池调度器。

它解决的核心问题是：**如何优雅地管理有限资源的分配与回收**。在 AI 编程工具的场景下，这个"有限资源"就是上游 API 的并发额度。

pool.js 提供三种使用方式：

**1. acquire + using：手动获取，自动释放**

```typescript
await using pool = new Pool({
    concurrency: 5,
    create: (i) => createDatabaseConnection(i),
});

using conn = await pool.acquire();
conn.value.query('SELECT * FROM users');
// 离开作用域自动释放
```

**2. Scheduler：自动调度，无需手动管理**

```typescript
const scheduler = pool.schedule();

const result = await scheduler.enqueue(
    function(this: Connection, query: string) {
        return this.query(query);
    },
    'SELECT * FROM users'
);
```

**3. 冷却回调：控制请求频率**

```typescript
const pool = new Pool({
    concurrency: 5,
    coolDown({ deliverAt, releaseAt }) {
        // 确保两次使用之间至少间隔 1 秒
        return new Promise(resolve => setTimeout(resolve, 1000));
    }
});
```

## 总结

AI 编程工具的限流问题，本质上是一个并发控制问题。与其在客户端做各种 hack，不如在中间加一层代理，统一控制节奏。

rate-proxy 就是这样一个方案：**一行命令启动，透明代理，零侵入**。

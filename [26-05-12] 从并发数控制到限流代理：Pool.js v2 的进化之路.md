从并发数控制到限流代理：Pool.js v2 的进化之路
========================================

自从上次写了一篇[《优雅完成高频面试题《请求并发数控制》》](./[23-12-08] 优雅完成高频面试题《请求并发数控制》.md)之后，时间过去了好一阵。当时那篇文章，从 PV 操作讲到资源池模式，最后提了一句我把它封装成了 npm 包 `@cat5th/pool.js`。

但是那个版本，说实话，写得很简陋。`acquire` 和 `release` 分开调用，使用者需要手动 `finally` 里释放资源，万一忘了，连接泄漏，线上事故。

最近我把整个库重写了一版，顺便做了一个基于它构建的限流 HTTP 代理工具：**rate-proxy**。今天就来聊聊这中间经历了什么。

## 旧版的问题

先回顾一下旧版的用法：

```typescript
const pool = new Pool([1, 2, 3, 4, 5]);

async function work() {
    const resource = await pool.acquire();
    try {
        await doSomething(resource);
    } finally {
        pool.release(resource); // 忘记写这行就泄漏了
    }
}
```

虽然用 `try...finally` 可以保证释放，但是：

- 使用者需要记住每次都要写 `finally`
- 整个池的清理也没有统一机制
- 资源创建和销毁的时机不好控制

这个问题，其实在我们日常开发中很常见。比如数据库连接池、Worker 线程池、甚至一个简单的 TCP 连接复用，都会遇到同样的问题。

## ES2024 显式资源管理

转机出现在 ES2024 引入了 **显式资源管理**（Explicit Resource Management），也就是 `using` 和 `await using` 关键字。

这个特性，和 C# 的 `using` 语句类似，当对象离开作用域时，会自动调用 `Symbol.dispose` 或 `Symbol.asyncDispose`。

```typescript
{
    using resource = acquireResource();
    doSomething(resource);
}
// 离开作用域后，resource[Symbol.dispose]() 自动被调用
```

这正好解决了我们旧版池管理的核心痛点：**资源释放不应该由使用者手动管理，而应该由语言层面保证它一定会被释放**。

于是，我把 pool.js 整个重写了。

## Pool.js v2

重写后的 v2 版本，核心变化就是拥抱 ES2024 的 `using` 关键字。

### 创建和使用

```typescript
import { Pool } from '@cat5th/pool.js';

// await using 池，离开作用域时自动清理
await using pool = new Pool({
    concurrency: 5,
    create: (i) => createDatabaseConnection(i),
});

// using 获取的资源，离开作用域时自动释放
using conn = await pool.acquire();
conn.value.query('SELECT * FROM users');

// 池离开作用域后，自动调用 Symbol.asyncDispose()
// 会等待所有已借出的资源归还后再清理
```

注意 `acquire()` 返回的是一个 `ResourceContainer`，实际资源要通过 `.value` 访问。容器实现了 `Symbol.dispose`，所以 `using` 能正确识别。

### 不手动调用 release

对比旧版：

```typescript
// 旧版：需要手动 release
using resource = await pool.acquire();  // 不，旧版没有 using
const conn = await pool.acquire();
try {
    // ...
} finally {
    pool.release(conn); // 容易忘
}
```

新版：

```typescript
// 新版：using 关键字确保释放
using conn = await pool.acquire();
// 离开作用域自动释放，不用管
```

### Scheduler：基于任务的调度

除了 `acquire` + `using` 的方式，v2 还提供了一个 `Scheduler`，让你完全不需要手动获取和释放资源。你只需要定义任务函数，Scheduler 会在有资源可用时自动调度：

```typescript
const scheduler = pool.schedule();

function work(this: Connection, query: string) {
    // this 就是池中的资源
    return this.query(query);
}

// 自动调度到可用资源
const result = await scheduler.enqueue(work, 'SELECT * FROM users');

// 或者包装成可复用的函数
const query = scheduler.wrap(work);
const rows = await query('SELECT 1');
```

这种方式下，你根本看不到 `acquire` 和 `release`，调度器在底层帮你管理资源的生命周期。

### CoolDown：冷却回调

另一个实用的功能是 `CoolDown`。资源被释放后，不一定能立即再次被使用。比如你对某个 API 做速率限制，需要保证两次请求之间至少间隔一定时间：

```typescript
const pool = new Pool({
    concurrency: 5,
    create: (i) => createHttpClient(i),
    coolDown({ deliverAt, releaseAt }) {
        // 确保两次使用之间至少间隔 1 秒
        const elapsed = releaseAt - deliverAt;
        const wait = Math.max(0, 1000 - elapsed);
        return new Promise(resolve => setTimeout(resolve, wait));
    }
});
```

`CoolDown` 回调会接收资源调度的时间信息，返回一个 Promise 来控制资源何时可以再次被分配。

## 实战：rate-proxy

光有库还不够，我基于 pool.js 做了一个实际可用的工具：**rate-proxy**。

它是一个带并发限制的 HTTP 反向代理服务器。使用场景很简单：你要代理某个上游 API，但不想让它被请求打垮。

```bash
npm install -g @cat5th/rate-proxy
TARGET=http://api.example.com CONCURRENCY=5 rate-proxy
```

核心实现其实就几行：

```typescript
import { Pool } from '@cat5th/pool.js';
import { createServer } from 'node:http';

const pool = new Pool(CONCURRENCY);
const scheduler = pool.schedule();

const server = createServer(async (req, res) => {
    await scheduler.enqueue(async function () {
        const response = await fetch(TARGET + req.url, {
            method: req.method,
            headers: req.headers,
            body: req,
        });
        res.writeHead(response.status);
        response.body.pipeTo(res);
    });
});

server.listen(PORT);
```

原理就是把 `scheduler.enqueue` 当作一个并发闸门。无论多少请求同时进来，只有 `CONCURRENCY` 个会真正到达上游，其余排队等待。

## 对比 p-limit

熟悉 Node.js 的同学可能用过 [p-limit](https://github.com/sindresorhus/p-limit)。pool.js v2 的 `Scheduler` 在行为上和它类似，但有两个关键区别：

**第一，pool.js 管理的是真实资源，而不仅仅是并发数。** p-limit 只控制并发数量，你拿到的是一个数字索引。pool.js 可以通过 `create` 工厂函数创建实际的资源对象（数据库连接、HTTP 客户端等），然后在任务函数中通过 `this` 直接使用。

**第二，pool.js 有完整的资源生命周期。** `Symbol.asyncDispose` 保证池被清理时，所有创建的资源都会被正确销毁。pre-created 资源（通过 `resources` 传入的）不会被销毁，适合外部传入的已有资源。

## 总结

从旧版的手动 `acquire` + `release`，到 v2 拥抱 ES2024 的 `using`，再到基于它构建的 rate-proxy，整个进化路径其实就围绕一个原则：

> 让正确的事情变得容易，让错误的事情变得不可能。

资源泄漏是一个很容易犯的错误，但通过语言层面的保证，使用者再也不用担心忘记释放资源。而 `Scheduler` 的存在，更是让你连 `using` 都不用写。
